Why Automate?

Manual processes are slow, inconsistent, and error-prone. Automation enables teams to:

🚀

Faster Delivery

Automated pipelines can run tests and deploy code in minutes, enabling multiple releases per day.

🔍

Catch Bugs Early

Automated tests run on every code change, catching regressions before they reach production.

🔁

Consistency & Repeatability

Automated processes run the same way every time, eliminating "works on my machine" problems.

🧑‍💻

Developer Productivity

Developers focus on writing code, not manual tasks. Automation handles repetitive operations.

The Automation Pyramid

      ▲ Manual Testing (least, most expensive)
    █████ UI / End-to-End Tests
  █████████ Integration Tests
█████████████ Unit Tests (most, fastest, cheapest)

The pyramid shape indicates the ideal proportion of tests. Many fast unit tests form the base; few slow end-to-end tests sit at the top.

Types of Testing

TypeWhat it TestsWho Runs ItSpeed
Unit Testing Individual functions/methods in isolation Developer Very fast (ms)
Integration Testing Interactions between components/modules Developer / QA Fast (seconds)
System Testing Complete system against requirements QA Slower (minutes)
Acceptance Testing (UAT) System meets user requirements Client / end users Slow (manual)
Regression Testing Existing functionality still works after changes Developer / CI Varies
Performance Testing Speed, stability, and scalability under load QA / DevOps Slow
Security Testing Vulnerabilities and weaknesses Security team Slow

White-Box vs Black-Box Testing

📋

Black-Box Testing

Tests functionality without knowledge of internal code. Focuses on inputs and expected outputs. Tests the "what", not the "how". Good for functional and acceptance testing.

🔬

White-Box Testing

Tests based on internal logic and code structure. Requires knowledge of the code. Ensures all code paths are exercised. Good for unit and integration testing.

⚖️

Grey-Box Testing

Partial knowledge of internals. Combines benefits of both approaches. Common in integration and API testing.

Test Coverage

Code coverage measures the percentage of source code executed during testing. High coverage doesn't guarantee quality — 100% coverage can still miss edge cases — but low coverage signals inadequate testing.

Statement coverage — every line executed at least once
Branch coverage — every true/false path taken
Path coverage — every unique path through code
Function coverage — every function called

Automated Testing

Automated tests are written as code. They run automatically, report pass/fail, and can be integrated into a CI/CD pipeline.

Unit Testing with pytest (Python)

Python — pytest Unit Tests 
# calculator.py  — the code being tested
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b
Python — pytest Test File 
# test_calculator.py
import pytest
from calculator import add, divide

def test_add_positive_numbers():
    assert add(2, 3) == 5

def test_add_negative_numbers():
    assert add(-1, -2) == -3

def test_add_zero():
    assert add(5, 0) == 5

def test_divide_normal():
    assert divide(10, 2) == 5.0

def test_divide_by_zero_raises_error():
    with pytest.raises(ValueError, match="Cannot divide by zero"):
        divide(10, 0)

# Run all tests: pytest test_calculator.py -v

Good Test Design: FIRST Principles

Fast — tests should run quickly
Independent — tests don't depend on each other
Repeatable — same result every time
Self-validating — pass or fail without manual inspection
Timely — written at the right time (before or with code)

TermDefinition

Test-Driven Development (TDD)

TDD
A development methodology where you write the test first, before writing any production code. Code is written only to make the test pass.

The Red-Green-Refactor Cycle

🔴

Red — Write a Failing Test

Write a test for the next small piece of functionality. The test must fail (because the code doesn't exist yet). This confirms the test is actually testing something.

🟢

Green — Make it Pass

Write the minimum amount of code to make the test pass. Don't over-engineer at this stage — just make it green.

🔵

Refactor — Improve the Code

Clean up the code while keeping all tests passing. Remove duplication, improve naming, apply design patterns. Tests give you confidence to refactor safely.

Benefits of TDD

  • Forces you to think about requirements before coding
  • Produces better-designed, more modular code
  • Creates a comprehensive test suite as a by-product
  • Provides confidence when refactoring
  • Reduces debugging time by catching errors immediately

CI/CD Pipelines

🔄

Continuous Integration (CI)

Developers frequently merge code changes to a shared repository. Each merge triggers an automated build and test suite. Catches integration errors quickly.

📦

Continuous Delivery (CD)

Automatically builds, tests, and prepares code for release. Code is always in a deployable state, but deployment to production is a manual step.

🚀

Continuous Deployment

Every change that passes all automated tests is automatically deployed to production. Requires very high test confidence.

Typical CI/CD Pipeline Stages

  1. 1

    Code Commit

    Developer pushes code to a version control system (Git). This triggers the pipeline.

  2. 2

    Build

    Source code is compiled/packaged. Dependencies are installed. A build artefact is produced.

  3. 3

    Automated Tests

    Unit tests, integration tests, and linting run. If any test fails, the pipeline stops and the developer is notified.

  4. 4

    Security Scanning

    Static analysis, dependency vulnerability checks (e.g., SAST tools, safety for Python).

  5. 5

    Deploy to Staging

    Artefact is deployed to a staging environment that mirrors production.

  6. 6

    Deploy to Production

    With Continuous Delivery, a human approves. With Continuous Deployment, this is fully automated.

GitHub Actions CI/CD Example

YAML — GitHub Actions Workflow (.github/workflows/ci.yml) 
name: CI Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.12'

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest pytest-cov

      - name: Run tests with coverage
        run: pytest --cov=src --cov-report=xml

      - name: Lint with flake8
        run: flake8 src/

DevOps

DevOps
A culture and set of practices that brings together software Development and IT Operations to shorten the development lifecycle and deliver high-quality software continuously.

Key DevOps Practices

Version control — all code (and infrastructure) in Git
CI/CD — automate build, test, deploy
Infrastructure as Code (IaC) — define servers with code (Terraform, Ansible)
Monitoring & logging — observe system behaviour in production
Feature flags — deploy code without exposing it to users immediately
Blue/Green deployment — run two production environments; switch traffic on successful deploy
Canary deployment — gradually roll out changes to a subset of users

DevOps vs Traditional Development

AspectTraditionalDevOps
Release frequencyMonthly/quarterlyDaily/multiple per day
Team structureSiloed Dev and Ops teamsCross-functional teams
TestingManual, at end of cycleAutomated, continuous
Feedback loopSlow (weeks/months)Fast (minutes/hours)
InfrastructureManual server configurationInfrastructure as Code

Scripting for Automation

Scripts are programs that automate repetitive tasks such as file management, data processing, deployments, and system maintenance.

Python Scripting

Python — Automation Script Example 
import os
import shutil
from datetime import datetime

def backup_files(source_dir, backup_dir):
    """Copy all .py files from source to timestamped backup directory."""
    timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
    dest = os.path.join(backup_dir, f"backup_{timestamp}")
    os.makedirs(dest, exist_ok=True)

    count = 0
    for filename in os.listdir(source_dir):
        if filename.endswith('.py'):
            shutil.copy(
                os.path.join(source_dir, filename),
                os.path.join(dest, filename)
            )
            count += 1

    print(f"Backed up {count} files to {dest}")
    return dest

backup_files("./src", "./backups")

Bash Scripting

Bash — Deployment Script 
#!/bin/bash
# deploy.sh — pull latest code, install deps, restart service

set -e  # Exit immediately on error

echo "=== Starting deployment ==="

# Pull latest code
git pull origin main

# Install/update dependencies
pip install -r requirements.txt

# Run database migrations
python manage.py migrate

# Run tests (abort deployment if they fail)
pytest tests/ -q

# Restart web server
systemctl restart myapp

echo "=== Deployment complete ==="

Containers & Deployment

Containers vs Virtual Machines

What it virtualisesEntire hardware (CPU, RAM, storage)OS user space (processes, files)
SizeGigabytesMegabytes
Boot timeMinutesSeconds or less
IsolationStrong (separate OS)Lighter (shares kernel)
PortabilityLess portableHighly portable ("build once, run anywhere")

Docker

Docker is the most popular container platform. A Dockerfile defines how to build a container image.

Dockerfile — Python Web App 
# Use official Python base image
FROM python:3.12-slim

# Set working directory
WORKDIR /app

# Install dependencies first (better caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY . .

# Expose the port the app runs on
EXPOSE 5000

# Run the application
CMD ["python", "app.py"]

Deployment Strategies

🔵🟢

Blue/Green Deployment

Two identical environments. New version deployed to "green" while "blue" is live. Traffic switched instantly. Easy rollback by switching back.

🐦

Canary Deployment

New version gradually rolled out to a small percentage of users. Monitor for errors before full rollout. Risk is limited to the canary group.

🔄

Rolling Deployment

Instances gradually replaced with the new version one by one. No downtime. Slower than blue/green; running two versions simultaneously during transition.

✅
Software Automation Key Takeaways
  • Unit tests test individual functions in isolation; they should be fast and numerous
  • CI automatically tests every code commit; CD automates deployment
  • TDD: write failing test → write code → refactor (Red-Green-Refactor)
  • DevOps breaks down silos between development and operations
  • Containers (Docker) ensure consistent environments across dev, staging, and production
  • Use deployment strategies (blue/green, canary) to minimise release risk
