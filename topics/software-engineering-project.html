Software Development Lifecycle (SDLC)

The SDLC is a structured process for planning, creating, testing, and deploying software. It provides a systematic approach to software development.

📋

1. Planning

Define project scope, objectives, resources, timeline, and feasibility. Identify stakeholders and risks.

🔍

2. Requirements Analysis

Gather and document what the system must do (functional requirements) and how it must perform (non-functional requirements).

✏️

3. Design

Architect the system: data structures, algorithms, interfaces, database schema, and technical stack.

💻

4. Implementation

Write the code according to design specifications. Follow coding standards; conduct code reviews.

🧪

5. Testing

Verify the system works as intended. Unit, integration, system, and acceptance testing.

🚀

6. Deployment & Maintenance

Release to production. Maintain, update, and improve the system over time. Handle bug reports.

Waterfall Model

Requirements → Design → Implementation → Testing → Deployment → Maintenance

Each phase must be fully completed before the next begins. Returns to a previous phase are costly. Suitable when requirements are well-understood and unlikely to change.

Spiral Model

Combines Waterfall's structure with iterative development. Each spiral loop covers: Planning → Risk Analysis → Engineering → Evaluation. Emphasises risk management. Good for large, complex, high-risk projects.

Scrum Framework

Scrum is the most widely used Agile framework. Work is organised into fixed-length iterations called sprints (typically 1–4 weeks).

Scrum Roles

Product Owner — represents stakeholders; manages and prioritises the Product Backlog; defines "what" to build
Scrum Master — facilitates Scrum process; removes impediments; coaches the team on Agile practices
Development Team — self-organising, cross-functional team of 3–9 people who build the product

Scrum Artefacts

Product Backlog — ordered list of all desired features/work items (user stories, bugs, tasks)
Sprint Backlog — subset of Product Backlog items selected for the current sprint
Increment — working, potentially shippable product at the end of each sprint

Scrum Ceremonies (Events)

User Stories

User Story
A brief description of a feature from the perspective of a user. Format: "As a [user type], I want [goal], so that [benefit]."

Examples:

  • "As a student, I want to search for notes by topic, so that I can quickly find what I need to study."
  • "As an admin, I want to reset user passwords, so that I can help locked-out users."

Kanban

Kanban is an Agile approach that visualises work on a board with columns (e.g., To Do → In Progress → Review → Done). Work in Progress (WIP) limits prevent overloading the team. Kanban is more flow-based than Scrum; there are no sprints.

Requirements Engineering

⚙️

Functional Requirements

What the system must do. Specific behaviours, features, and functions. Example: "The system must allow users to register with an email and password."

📊

Non-Functional Requirements

How the system must perform. Quality attributes. Examples: "The login page must load in under 2 seconds." "The system must be available 99.9% of the time."

Non-Functional Requirement Categories

Performance — response time, throughput, latency
Scalability — ability to handle growth in users/data
Reliability — uptime, fault tolerance, error recovery
Security — authentication, authorisation, encryption
Usability — ease of use, accessibility
Maintainability — ease of updating and debugging
Portability — ability to run on different platforms

Requirements Elicitation Techniques

  • Interviews — structured conversations with stakeholders
  • Surveys / Questionnaires — gather input from many stakeholders efficiently
  • Workshops — collaborative sessions to define requirements
  • Observation — watch users perform current tasks
  • Prototyping — build mockups to validate requirements with users
  • Document analysis — review existing systems, processes, and documentation

Use Case Diagrams

Use case diagrams (a type of UML diagram) show the interactions between actors (users or external systems) and the system's use cases (functions). They provide a high-level view of system functionality.

Actor — a role that interacts with the system (stick figure)
Use case — a function or feature of the system (oval)
Association — line connecting actor to use case
Include — a use case always includes another (dashed arrow with «include»)
Extend — a use case optionally extends another (dashed arrow with «extend»)
System boundary — rectangle enclosing all use cases

System Design

Data Flow Diagrams (DFDs)

DFDs show how data flows through a system — from inputs, through processes, to outputs and data stores. They have different levels of detail:

Context diagram (Level 0) — single process showing the whole system and external entities
Level 1 DFD — breaks down the main process into sub-processes
Level 2 DFD — further detail on each sub-process
External entity — box (sources/destinations of data outside the system)
Process — rounded rectangle (transforms data)
Data store — open-ended rectangle (stored data)
Data flow — arrow (movement of data)

Structure Charts

Structure charts show the hierarchical decomposition of a program into modules and sub-modules. They illustrate the call hierarchy (which modules call which) and data coupling (data passed between modules).

Database Design — ERDs

An Entity-Relationship Diagram (ERD) models the entities in a system and the relationships between them. Used to design relational database schemas.

Entity — a real-world object (table in a database)
Attribute — a property of an entity (column)
Primary key — unique identifier for each record
Foreign key — attribute that references a primary key in another table
Relationship — association between two entities (one-to-one, one-to-many, many-to-many)

Algorithm Design

Algorithms should be designed before coding. Common tools:

Pseudocode — Binary Search 
FUNCTION binary_search(sorted_list, target):
    left ← 0
    right ← length(sorted_list) - 1

    WHILE left ≤ right:
        mid ← (left + right) DIV 2
        IF sorted_list[mid] = target THEN
            RETURN mid
        ELSE IF sorted_list[mid] < target THEN
            left ← mid + 1
        ELSE
            right ← mid - 1

    RETURN -1  // not found

Version Control with Git

Version Control System (VCS)
A system that tracks changes to files over time. Allows teams to collaborate, revert to previous versions, and maintain a complete history of changes. Git is the most widely used VCS.

Key Git Commands

Git — Essential Commands 
# Initialise a repository
git init

# Clone a remote repository
git clone https://github.com/user/repo.git

# Check status of working directory
git status

# Stage changes for commit
git add filename.py      # specific file
git add .               # all changes

# Commit staged changes
git commit -m "Add user authentication feature"

# Push to remote repository
git push origin main

# Pull latest changes
git pull origin main

# Create and switch to a new branch
git checkout -b feature/user-login

# Merge a branch into main
git checkout main
git merge feature/user-login

# View commit history
git log --oneline

Git Branching Strategies

🌿

Feature Branching

Each new feature is developed on its own branch. Merged into main when complete via a pull request. Keeps main always stable.

🌊

Git Flow

Uses main, develop, feature/*, release/*, and hotfix/* branches. Strict but provides clear structure for larger teams.

🔀

Pull Requests (PRs)

A request to merge one branch into another. Provides a chance for code review, discussion, and automated tests before merging.

Collaboration & Code Review

  • Code review — peers examine code for bugs, security issues, style violations, and improvements before merging
  • Issues/Tickets — track bugs and features in tools like GitHub Issues or Jira
  • Commit messages — should be clear and descriptive: "Fix login bug when email contains uppercase letters"
  • .gitignore — lists files that should not be tracked (secrets, build artefacts, dependencies)

Python — Well-documented function with docstring 
def calculate_bmi(weight_kg: float, height_m: float) -> float:
    """
    Calculate Body Mass Index (BMI).

    Args:
        weight_kg: Body weight in kilograms (must be positive).
        height_m: Height in metres (must be positive).

    Returns:
        float: The BMI value rounded to 2 decimal places.

    Raises:
        ValueError: If weight or height is not positive.

    Example:
        >>> calculate_bmi(70, 1.75)
        22.86
    """
    if weight_kg <= 0 or height_m <= 0:
        raise ValueError("Weight and height must be positive")
    return round(weight_kg / (height_m ** 2), 2)

Testing Strategy

Test Plan

A test plan documents the overall approach to testing a project. It includes:

  • Scope — what will and won't be tested
  • Testing types to be performed
  • Test environment and tools
  • Test schedule and resources
  • Pass/fail criteria

Test Cases

Boundary Value Analysis

Test values at and just inside/outside the boundaries of input ranges. Errors often occur at boundaries. For a field accepting ages 18–65, test: 17, 18, 19, 64, 65, 66.

Equivalence Partitioning

Divide input data into classes where the system should behave the same way. Test one value from each partition. For a password field (8–20 chars): test too short (5), valid (12), too long (25).

Project Evaluation

Evaluation determines how well the completed project meets its objectives and requirements. It is an essential part of the SE project assessment.

What to Evaluate

Functionality — does it do what it was supposed to?
Correctness — does it produce correct results?
Usability — how easy is it for users to use?
Performance — is it fast enough?
Reliability — does it work consistently without errors?
Security — are appropriate measures in place?
Maintainability — how easy would it be to extend or modify?

Evaluation Against Requirements

For each requirement documented in the planning phase, state whether it was met and provide evidence:

RequirementMet?Evidence
Users can register with email/password✅ YesTest case TC-001 passes; screenshot of registration flow
Login page loads in < 2 seconds✅ YesPerformance test shows average 850ms load time
Password reset via email⚠️ PartialFunctionality works but email delivery is slow in testing

Reflection and Improvements

A strong evaluation includes honest reflection on:

  • Challenges encountered and how they were addressed
  • What could be improved with more time/resources
  • What was learned during the project
  • How the methodology worked and what could be done differently
✅
SE Project Key Takeaways
  • Agile is iterative; Waterfall is sequential
  • Scrum: Product Backlog → Sprint Planning → Sprint → Review → Retrospective
  • Functional requirements = what it does; Non-functional = how well it does it
  • Git: commit early and often; use branches for features
  • Document your design decisions and testing evidence
  • Evaluation must compare outcomes against your stated requirements
