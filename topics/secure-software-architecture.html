The CIA Triad

The CIA triad is the foundational model of information security. Every security decision should be evaluated against its three principles.

🔐

Confidentiality

Ensuring that information is accessible only to those authorised to have access. Achieved through encryption, access controls, and authentication.

✅

Integrity

Maintaining the accuracy and completeness of data. Data must not be altered by unauthorised parties. Achieved via checksums, hashing, and digital signatures.

🟢

Availability

Ensuring that authorised users have reliable access to systems and data when needed. Protected through redundancy, backups, and DDoS mitigation.

📝
Exam Tip

When answering exam questions about security, always consider which aspect(s) of the CIA triad are relevant. A breach that leaks data violates Confidentiality; one that corrupts data violates Integrity; one that crashes a server violates Availability.

Additional Security Principles

Non-repudiation — inability to deny an action (digital signatures provide this)
Least Privilege — users/processes should have only the permissions they need
Defence in Depth — multiple overlapping layers of security controls
Security by Design — security built in from the start, not added later
Fail-Safe Defaults — deny access by default; only allow what is explicitly permitted
Open Design — security should not rely on secrecy of the mechanism itself (Kerckhoffs's principle)

STRIDE Threat Model

STRIDE is a systematic framework developed by Microsoft for identifying security threats:

🥸

Spoofing

Pretending to be someone else. Violates Authentication.

✏️

Tampering

Modifying data maliciously. Violates Integrity.

🙈

Repudiation

Denying actions were taken. Violates Non-repudiation.

👁️

Information Disclosure

Exposing data to unauthorised parties. Violates Confidentiality.

💥

Denial of Service

Making a system unavailable. Violates Availability.

🔑

Elevation of Privilege

Gaining higher access than permitted. Violates Authorisation.

Authentication & Authorisation

ℹ️
Authentication vs Authorisation

Authentication (AuthN) — verifying who you are (identity). Authorisation (AuthZ) — determining what you are allowed to do (permissions).

Authentication Factors

Strong authentication combines multiple factor types:

Something you know — passwords, PINs, security questions
Something you have — hardware token, smart card, phone (OTP app)
Something you are — biometrics: fingerprint, face, retina scan
Somewhere you are — geolocation (less common)

Multi-Factor Authentication (MFA)

MFA requires two or more factors from different categories. Even if a password is compromised, a second factor (e.g., an OTP sent to a phone) prevents unauthorised access. Two-Factor Authentication (2FA) is the most common form.

Password Security

Never store passwords in plaintext — always hash them
Use a slow hashing algorithm: bcrypt, argon2, or scrypt
Add a unique random salt before hashing to prevent rainbow table attacks
Enforce minimum length (≥ 12 chars) and complexity
Implement account lockout after failed attempts
Python — Secure Password Hashing with bcrypt 
import bcrypt

# Hashing a password (salt is generated automatically)
password = "MySecurePassword123!".encode('utf-8')
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=12))

# Verifying a password
def verify_password(plain_password, hashed_password):
    return bcrypt.checkpw(
        plain_password.encode('utf-8'),
        hashed_password
    )

# NEVER do this:
# db.save(password=plain_text_password)  ← INSECURE!

Authorisation Models

ModelDescriptionExample
Role-Based Access Control (RBAC) Permissions assigned to roles; users assigned to roles Admin, Editor, Viewer roles in a CMS
Attribute-Based Access Control (ABAC) Access based on attributes of user, resource, and environment A doctor can view records only for their patients
Discretionary Access Control (DAC) Resource owners control access to their resources Unix file permissions (owner, group, other)
Mandatory Access Control (MAC) System enforces access based on security labels Military classification levels (Secret, Top Secret)

Encryption

Encryption transforms readable data (plaintext) into an unreadable format (ciphertext) using an algorithm and a key. Only authorised parties with the correct key can decrypt it.

🔑

Symmetric Encryption

Same key is used for both encryption and decryption. Fast and efficient for large data. Key distribution is a challenge. Examples: AES, DES (deprecated), 3DES.

🗝️

Asymmetric Encryption

Two mathematically related keys: a public key (share freely) and a private key (keep secret). Slower than symmetric. Examples: RSA, ECC, DSA.

#️⃣

Hashing

One-way transformation producing a fixed-length digest. Cannot be reversed. Used for password storage and integrity checks. Examples: SHA-256, SHA-3, MD5 (deprecated).

⚠️
Hashing is NOT Encryption

Hashing is a one-way process — you cannot "decrypt" a hash. Encryption is two-way. Do not use hashing for data you need to recover; use it for passwords and integrity verification only.

How Asymmetric Encryption Works

  1. 1

    Key Generation

    Alice generates a key pair: one public key and one private key.

  2. 2

    Public Key Distribution

    Alice shares her public key widely (e.g., on a certificate authority).

  3. 3

    Encryption

    Bob encrypts a message using Alice's public key.

  4. 4

    Decryption

    Only Alice's private key can decrypt the ciphertext. No one else can read it.

Digital Signatures

Digital signatures use asymmetric encryption in reverse to provide authentication and non-repudiation:

  • The sender signs a message by encrypting its hash with their private key.
  • Anyone with the sender's public key can verify the signature.
  • If the signature is valid, the message is authentic and hasn't been tampered with.

Hybrid Encryption

In practice, asymmetric encryption is used to securely exchange a session key, which is then used for fast symmetric encryption of the actual data. This is how TLS/HTTPS works.

Security Protocols

SSL / TLS

Transport Layer Security (TLS) — and its predecessor SSL — provides encrypted, authenticated communication over a network. HTTPS = HTTP + TLS.

TLS Handshake — establishes a shared session key using asymmetric encryption
Certificate Authority (CA) — trusted third party that issues and validates digital certificates
Certificate — binds a public key to an identity; signed by a CA
HTTPS — HTTP secured with TLS; protects confidentiality and integrity

Secure Coding Practices

Secure code is not added at the end — it must be built into every stage of development.

Input Validation

Never trust input from users or external systems. Always validate and sanitise input before use.

🚨
The #1 Rule: Never Trust User Input

Validate on both the client side (UX) and the server side (security). Client-side validation can always be bypassed.

SQL Injection Prevention

Python — Vulnerable vs Secure SQL 
# ❌ VULNERABLE — string concatenation opens SQL injection
username = request.form['username']
query = f"SELECT * FROM users WHERE username = '{username}'"
cursor.execute(query)  # Attacker can enter: ' OR 1=1 --

# ✅ SECURE — parameterised queries prevent SQL injection
username = request.form['username']
query = "SELECT * FROM users WHERE username = ?"
cursor.execute(query, (username,))  # Input is treated as data, not code

Cross-Site Scripting (XSS) Prevention

Python — XSS Prevention via HTML Escaping 
import html

# ❌ VULNERABLE — raw user input rendered in HTML
user_comment = '<script>document.cookie = "stolen"</script>'
html_output = f"<p>{user_comment}</p>"  # Script will execute!

# ✅ SECURE — escape special HTML characters
safe_comment = html.escape(user_comment)
html_output = f"<p>{safe_comment}</p>"
# Output: <p>&lt;script&gt;...&lt;/script&gt;</p>

Key Secure Coding Principles

Parameterised Queries — prevent SQL injection
Output Encoding — prevent XSS by encoding user data in output
Input Validation — whitelist acceptable values; reject everything else
Error Handling — never expose stack traces or system details to users
Principle of Least Privilege — database accounts should have minimal permissions
Secure Dependencies — keep libraries and frameworks up to date; audit regularly
Secrets Management — never hardcode passwords, API keys, or tokens in source code
HTTPS Everywhere — always use TLS; redirect HTTP to HTTPS

Network Security

🧱

Firewall

Monitors and controls incoming/outgoing network traffic based on predefined rules. Can be hardware- or software-based. Types: packet filtering, stateful inspection, application-layer (WAF).

🕵️

Intrusion Detection System (IDS)

Monitors network/system activity for malicious behaviour and policy violations. Raises alerts when suspicious activity is detected. Detects but does not prevent.

🛡️

Intrusion Prevention System (IPS)

Like an IDS but actively blocks detected threats. Sits inline on the network. Can also be combined: IDPS.

🌐

DMZ (Demilitarised Zone)

A network segment that sits between the internal network and the internet. Public-facing servers (web, email) are placed here to limit exposure of the internal network.

🔒

VPN

Creates an encrypted tunnel over a public network. Allows remote workers to securely access internal resources. Uses protocols like OpenVPN, IPSec, or WireGuard.

🔍

Network Monitoring

Continuously observing network traffic for anomalies, intrusions, and performance issues. Tools include Wireshark, Nagios, and SIEM platforms.

Defence in Depth

Security should not rely on a single control. Multiple layers of protection are required:

[ Internet ]
↓ Firewall / WAF
↓ DMZ (Web Servers, Email Servers)
↓ Internal Firewall
↓ IDS / IPS
↓ Internal Network (Application Servers)
↓ Database Firewall
↓ [ Database ]

Social Engineering

Social engineering exploits human psychology rather than technical vulnerabilities. It is often the easiest way to breach an organisation's security.

⚠️
The Weakest Link

"The weakest link in the security chain is the human element." — Kevin Mitnick. Technical controls are effective only when humans are also educated and vigilant.

TechniqueDescriptionPrevention
Phishing Mass email impersonating legitimate organisations to steal credentials Security awareness training; email filtering; MFA
Spear Phishing Targeted phishing using personalised information about the victim Employee training; verify email senders; DMARC/SPF records
Vishing Voice phishing — phone calls impersonating IT support, banks, etc. Verification procedures; never give credentials over phone
Pretexting Creating a fabricated scenario to extract information Strict identity verification; need-to-know policies
Baiting Leaving infected USB drives in public places Disable auto-run; prohibit unknown USB devices
Tailgating Following an authorised person through a secure door Access cards; security guards; mantraps

Privacy & Legislation

Australian Privacy Principles (APPs)

The Privacy Act 1988 (Cth) regulates the handling of personal information by Australian Government agencies and private sector organisations. It contains 13 Australian Privacy Principles covering:

  • Open and transparent management of personal information
  • Anonymity and pseudonymity
  • Collection of solicited and unsolicited personal information
  • Dealing with information: use, disclosure, and quality
  • Security and access to personal information
  • Cross-border disclosure of personal information

Notifiable Data Breaches (NDB) Scheme

Organisations covered by the Privacy Act must notify the Office of the Australian Information Commissioner (OAIC) and affected individuals when a data breach is likely to cause serious harm.

GDPR (General Data Protection Regulation)

The EU's GDPR applies to any organisation handling EU residents' data. Key principles:

Lawfulness, fairness, and transparency
Purpose limitation — data collected for specific, stated purposes only
Data minimisation — collect only what is necessary
Right to be forgotten — users can request deletion of their data
Breach notification — within 72 hours

Ethical Responsibilities of Software Engineers

Developers have ethical and professional obligations regarding user data:

  • Collect only the minimum data necessary (data minimisation)
  • Be transparent about what data is collected and why
  • Obtain informed consent before collecting sensitive data
  • Implement appropriate security controls to protect user data
  • Promptly disclose and remediate security vulnerabilities
  • Consider accessibility and inclusivity in security design

Security Testing

🔍

Vulnerability Scanning

Automated tools scan systems for known vulnerabilities. Examples: Nessus, OpenVAS. Quick but may produce false positives.

⚔️

Penetration Testing (Pen Test)

Authorised simulated attack on a system to find exploitable vulnerabilities. Performed by ethical hackers (white-hat). Types: black-box, white-box, grey-box.

🔎

Code Review

Manual or automated inspection of source code for security flaws. Static Application Security Testing (SAST) tools automate this.

🌊

Fuzzing

Sending random, invalid, or unexpected inputs to a program to trigger crashes or unexpected behaviour that reveals vulnerabilities.

Pen Testing Types

TypeTester's KnowledgeUse Case
Black-boxNo internal knowledge (simulates external attacker)Testing external defences
White-boxFull knowledge of architecture and source codeThorough internal review
Grey-boxPartial knowledge (simulates insider or authenticated user)Realistic threat scenarios

Summary: Key Security Concepts to Remember
  • CIA Triad: Confidentiality, Integrity, Availability
  • Symmetric encryption uses one key; Asymmetric uses a key pair
  • Hashing is one-way — used for passwords and integrity
  • MFA combines multiple authentication factors
  • Defence in Depth uses multiple security layers
  • OWASP Top 10 lists the most critical web vulnerabilities
  • Never trust user input — always validate and sanitise
